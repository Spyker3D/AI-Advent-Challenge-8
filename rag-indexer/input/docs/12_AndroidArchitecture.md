## Зачем нужна архитектура

Хорошая архитектура делает приложение проще для сопровождения, тестирования и масштабирования. Каждый слой отвечает только за свою задачу и минимально зависит от остальных.

---

## Слои приложения

Современное Android-приложение обычно состоит из трёх слоёв:

- UI Layer
- Domain Layer
- Data Layer

Такое разделение соответствует принципу разделения ответственности (Separation of Concerns).

---

## Общая схема

```text
Jetpack Compose UI
        │
        ▼
    ViewModel
        │
        ▼
 Use Cases (Domain)
        │
        ▼
   Repository
        │
        ▼
Retrofit / Room / DataStore
        │
        ▼
 OpenRouter / База данных / API
```

---

## UI Layer

Слой пользовательского интерфейса включает:

- Jetpack Compose;
- ViewModel;
- управление состоянием экрана;
- обработку действий пользователя.

UI не обращается напрямую к сети или базе данных.

---

## Domain Layer

Domain Layer содержит бизнес-логику приложения.

Основные элементы:

- Use Cases;
- модели предметной области;
- интерфейсы репозиториев.

Этот слой не зависит от Android Framework, поэтому его легко тестировать.

---

## Data Layer

Data Layer отвечает за получение и хранение данных.

Компоненты:

- Repository;
- Retrofit;
- Room;
- DataStore;
- удалённые API;
- локальное хранилище.

Именно здесь происходит выбор источника данных.

---

## Поток данных

Типичный сценарий работы:

1. Пользователь отправляет сообщение.
2. ViewModel вызывает Use Case.
3. Use Case обращается к Repository.
4. Repository получает данные через Retrofit или локальное хранилище.
5. Результат возвращается обратно в ViewModel.
6. UI автоматически обновляется.

---

## Структура проекта

```text
app/
 ├── ui/
 ├── viewmodel/
 ├── domain/
 │    ├── model/
 │    ├── repository/
 │    └── usecase/
 ├── data/
 │    ├── remote/
 │    ├── local/
 │    └── repository/
 ├── di/
 └── util/
```

Такая структура остаётся понятной даже при росте проекта.

---

## Best Practices

- Разделяйте ответственность между слоями.
- Не обращайтесь к сети напрямую из UI.
- Используйте Repository как единую точку доступа к данным.
- Храните бизнес-логику в Domain Layer.
- Внедряйте зависимости через Dependency Injection.

---

# Итоги

Многослойная архитектура делает Android AI Assistant более устойчивым к изменениям, облегчает повторное использование кода и упрощает тестирование. Совместное использование MVVM, Repository, Retrofit, Room и Dependency Injection позволяет построить современное и масштабируемое приложение.
