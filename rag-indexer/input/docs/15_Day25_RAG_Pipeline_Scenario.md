# RAG Pipeline Implementation in Android AI Assistant

> Документ описывает реализацию Retrieval-Augmented Generation (RAG) в Android AI Assistant и предназначен для использования в базе знаний проекта.

---

# 1. Назначение RAG

Android AI Assistant использует Retrieval-Augmented Generation (RAG), чтобы отвечать пользователю не только на основе знаний языковой модели, но и на основе локальной базы знаний проекта.

Перед генерацией ответа выполняется поиск наиболее релевантных документов, которые затем передаются в LLM в качестве дополнительного контекста.

Это позволяет:

- уменьшить количество галлюцинаций;
- отвечать с учетом архитектуры проекта;
- использовать актуальную документацию;
- указывать источники и цитаты.

---

# 2. Общий RAG Pipeline

Полный процесс обработки пользовательского вопроса выглядит следующим образом.

```text
User Question
        │
        ▼
Task Memory
        │
        ▼
Recent Chat History
        │
        ▼
Query Rewrite (optional)
        │
        ▼
Embedding Model
        │
        ▼
Retriever
        │
        ▼
Similarity Filter
        │
        ▼
Hybrid Reranker
        │
        ▼
Confidence Check
        │
        ▼
Prompt Builder
        │
        ▼
LLM
        │
        ▼
Assistant Answer
        │
        ▼
Task Memory Update
```

---

# 3. Query Rewrite

Перед поиском пользовательский вопрос может быть автоматически переписан.

Цель Query Rewrite:

- убрать неоднозначность;
- добавить технические термины;
- повысить качество поиска;
- не изменить смысл вопроса.

При этом используются два варианта запроса:

- **Original Question** — отображается пользователю и передается в финальный prompt;
- **Rewritten Query** — используется только для поиска по базе знаний.

Таким образом пользователь всегда видит именно свой исходный вопрос.

---

# 4. Embedding Model

После Query Rewrite создается embedding поискового запроса.

Embedding представляет собой вектор, отражающий смысл вопроса.

Для каждого chunk базы знаний embedding был создан заранее во время индексации.

---

# 5. Retrieval

Retriever выполняет поиск наиболее релевантных chunks.

Поиск осуществляется по embedding пользователя и embedding каждого chunk.

В качестве основной метрики используется косинусное сходство (Cosine Similarity).

Retriever возвращает Top-K наиболее похожих chunks.

---

# 6. Similarity Filter

После Retrieval найденные chunks проходят дополнительную фильтрацию.

Chunks с низким Cosine Similarity исключаются.

Если после фильтрации остается слишком мало результатов, используется fallback и выбираются наиболее релевантные chunks из первоначального списка.

Такой подход уменьшает количество случайных совпадений.

---

# 7. Hybrid Reranker

После первичного поиска найденные chunks дополнительно переоцениваются.

Для ранжирования используется несколько факторов.

## Cosine Score

Оценивает смысловую близость вопроса и chunk с помощью embeddings.

## Keyword Score

Учитывает совпадения важных терминов между вопросом пользователя и найденным chunk.

## Metadata Score

Использует дополнительную информацию:

- название документа;
- раздел документа;
- источник;
- другие метаданные.

Финальный рейтинг вычисляется по формуле:

```text
finalScore =
0.70 × cosineScore +
0.20 × keywordScore +
0.10 × metadataScore
```

После пересчета выбираются наиболее релевантные chunks.

---

# 8. Confidence Check

Перед отправкой запроса в LLM оценивается качество Retrieval.

Для оценки используется лучший найденный chunk.

Если confidence ниже установленного порога, ассистент не отправляет запрос в LLM.

Вместо этого пользователю предлагается уточнить вопрос.

Такой механизм уменьшает вероятность генерации недостоверной информации.

---

# 9. Prompt Builder

Prompt Builder формирует итоговый prompt для языковой модели.

Он объединяет:

- системные инструкции;
- Task Memory;
- последние сообщения диалога;
- найденный RAG-контекст;
- исходный вопрос пользователя.

Каждый найденный chunk содержит:

- Source;
- Section;
- Chunk ID;
- Quote.

Таким образом модель отвечает исключительно на основе найденного контекста.

---

# 10. Источники и цитаты

Каждый ответ ассистента должен содержать:

- основной ответ;
- список использованных источников;
- разделы документов;
- Chunk ID (если доступен);
- цитаты из найденных документов.

Это позволяет проверить происхождение каждого факта и уменьшает вероятность галлюцинаций.

---

# 11. Task Memory

Во время диалога используется Task Memory.

Task Memory хранит информацию о текущей задаче пользователя.

Она может содержать:

- цель диалога;
- ограничения;
- принятые решения;
- текущее состояние задачи;
- важные уточнения;
- ключевые технические термины.

Task Memory помогает корректно интерпретировать короткие вопросы, например:

- "А дальше?"
- "Почему?"
- "Как это сделать?"
- "Что происходит после этого?"

При этом Task Memory **не заменяет Retrieval**.

При каждом новом вопросе поиск по базе знаний выполняется заново.

---

# 12. Recent Chat History

Помимо Task Memory в prompt добавляется ограниченная история последних сообщений.

Она позволяет сохранять контекст текущего разговора без передачи всей истории диалога.

Обычно используется несколько последних сообщений или ограничение по размеру текста.

---

# 13. Преимущества реализации

Использование RAG позволяет:

- использовать знания проекта при генерации ответов;
- уменьшить количество галлюцинаций;
- отвечать с учетом документации;
- всегда указывать источники информации;
- поддерживать длинные технические диалоги благодаря Task Memory и истории сообщений.

---

# 14. Итог

RAG в Android AI Assistant объединяет несколько компонентов:

- Query Rewrite;
- Embedding Model;
- Retriever;
- Similarity Filter;
- Hybrid Reranker;
- Confidence Check;
- Prompt Builder;
- Task Memory;
- Recent Chat History;
- Large Language Model.

Совместная работа этих компонентов позволяет получать более точные, объяснимые и воспроизводимые ответы на основе локальной базы знаний проекта.
