# 1. Назначение модуля

Chat Module — основной модуль приложения.

Именно через него пользователь взаимодействует с AI.

Основные задачи:

- отображение истории сообщений;
- ввод нового сообщения;
- отправка сообщения модели;
- отображение streaming-ответа;
- обработка ошибок;
- отображение состояния загрузки;
- поддержка будущих AI-инструментов.

# 4. Роли сообщений

В AI-чатах обычно используются четыре роли.

## System

Определяет поведение модели.

Пример:

```text
Ты Android AI Assistant.
```

---

## User

Сообщение пользователя.

Пример:

```text
Что такое Jetpack Compose?
```

---

## Assistant

Ответ модели.

Пример:

```text
Jetpack Compose — современный UI Toolkit...
```

---

## Tool

Используется при работе Function Calling и MCP.

Например:

```text
Weather Tool
Calculator Tool
File Search
```

# 6. Отправка сообщения

Алгоритм отправки.

1.

Пользователь вводит текст.

```
Привет!
```

2.

Нажимает кнопку.

3.

ViewModel проверяет:

- не пустой ли текст;
- нет ли уже выполняющегося запроса.

4.

Создается сообщение пользователя.

5.

Оно сразу отображается в UI.

6.

Начинается запрос к AI.

7.

После получения ответа добавляется сообщение Assistant.

# 7. ChatUiState

В Compose экран обычно управляется одним состоянием.

```kotlin
data class ChatUiState(

    val messages: List<ChatMessageUi> = emptyList(),

    val input: String = "",

    val isLoading: Boolean = false,

    val error: String? = null
)
```

---

## Почему единое состояние?

Compose любит immutable state.

Вместо:

```kotlin
var loading

var input

var messages
```

лучше использовать

```kotlin
ChatUiState
```

# 8. История сообщений

Каждый новый запрос должен учитывать предыдущие сообщения.

Например

```text
User

Что такое Compose?

Assistant

Compose — UI Toolkit.

User

А кто его разработал?
```

Второй вопрос нельзя отправить отдельно.

Модель не поймет контекст.

Поэтому Repository отправляет историю.

---

## Пример

```json
[
    {
        "role":"user",
        "content":"Что такое Compose?"
    },
    {
        "role":"assistant",
        "content":"Compose — UI Toolkit."
    },
    {
        "role":"user",
        "content":"Кто его разработал?"
    }
]
```

---

# 9. Ограничение контекста

У моделей есть ограничение по количеству токенов.

Поэтому нельзя хранить бесконечную историю.

Возможные стратегии:

- последние N сообщений;
- суммаризация старых сообщений;
- RAG;
- Memory.

# 10. Streaming

Streaming делает чат более отзывчивым.

Без streaming пользователь ждет:

```
...
...
...
готовый ответ
```

Со streaming:

```
Jet...

Jetpack...

Jetpack Compose...
```

---

## Интерфейс Repository

```kotlin
interface ChatRepository {

    fun streamMessage(

        message: String

    ): Flow<String>

}
```

---

## ViewModel

```kotlin
viewModelScope.launch {

    repository

        .streamMessage(message)

        .collect {

            appendAssistantChunk(it)

        }

}
```

# 11. Добавление сообщения в список

Обычно происходит так.

Пользователь:

```
Привет
```

↓

Добавляем сообщение пользователя.

↓

Добавляем пустое сообщение Assistant.

↓

Начинаем постепенно обновлять его содержимое.

Это избавляет от постоянного создания новых объектов.

# 12. Обработка ошибок

Типичные ошибки:

- нет интернета;
- отмена запроса;
- timeout;
- превышение лимита;
- неправильный API Key;
- ошибка сервера.

Не рекомендуется показывать пользователю stacktrace.

Лучше использовать понятные сообщения.

---

# 13. Retry

Хорошей практикой является возможность повторной отправки.

Например.

```
Ошибка сети

[ Повторить ]
```

В этом случае используется последнее сообщение пользователя.


# 15. Prompt Builder

Формирование промпта рекомендуется вынести отдельно.

Плохо:

```kotlin
repository.sendMessage(...)
```

Хорошо:

```text
ViewModel

↓

PromptBuilder

↓

Repository
```

PromptBuilder отвечает за:

- system prompt;
- историю;
- RAG;
- MCP;
- tool results.

---

# 16. Подготовка к MCP

Когда пользователь пишет

```
Какая сегодня погода?
```

Repository должен определить,

нужно ли обращаться к Tool.

В будущем схема станет такой.

```text
User

↓

Prompt Builder

↓

Tool Router

↓

Weather Tool

↓

LLM
```

# 17. Подготовка к локальным моделям

Repository не должен зависеть от OpenAI.

Лучше использовать интерфейс.

```kotlin
interface AiProvider {

    suspend fun chat(...)

}
```

Реализации:

- OpenAI
- Ollama
- Gemini
- Claude

# 20. Best Practices

✅ UI ничего не знает про API.

✅ ViewModel не знает про Retrofit.

✅ Repository не знает про Compose.

✅ История хранится отдельно от UI.

✅ Prompt Builder — отдельный класс.

✅ Streaming использовать через Flow.

✅ Ошибки типизировать.

✅ Подготовить модуль к RAG и MCP заранее.

---

# Заключение

Chat Module является центральной частью Android AI Assistant.

Именно здесь объединяются пользовательский интерфейс, работа с LLM, история сообщений, потоковая генерация ответов и будущие механизмы RAG и MCP.

Архитектура должна оставаться максимально независимой от конкретного AI-провайдера, чтобы в дальнейшем можно было легко заменить OpenAI на Ollama, Claude, Gemini или локальную модель без изменения пользовательского интерфейса.
